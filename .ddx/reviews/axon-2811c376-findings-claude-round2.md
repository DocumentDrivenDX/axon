## Findings

| Severity | Area | Finding |
|---|---|---|
| WARNING | Format tightening scope | Enabling `should_validate_formats(true)` is a **workspace-global behavior change** applied to every existing `validate_entity` call, not just the three formats the bead names. The codebase currently has 3 `format` uses, all `date-time` (`crates/axon-schema/src/named_queries.rs:568`, `crates/axon-api/src/handler.rs:16704`, `crates/axon-schema/fixtures/beads.esf.yaml:53`), and jsonschema's `date-time` assertion is strict RFC3339. The plan never audits whether the data currently validated against those `date-time` fields is RFC3339-valid â€” it relies on the full `cargo test` gate (item 8) to surface breakage late. Any timestamp string the system emits without a timezone offset will now fail validation that previously passed. The plan should enumerate these three sites and confirm their data, not discover failures at the gate. |
| WARNING | Scope creep beyond bead | Section 6 extends format assertion to `validate_link_metadata` ("same... format behavior") and to `validate_entity`. The bead requires `email`/`uuid`/`date-time` assertion; it does not request tightening link-metadata validation. This is an unrequested behavior change to a second code path (`crates/axon-schema/src/validation.rs:139`) whose existing error format (joined-string with `instance_path` prefix, lines 145-163) differs from `validate_entity`'s structured errors. The plan does not specify how the owned `SchemaValidationErrors` map back into that joined-string `AxonError` shape. |
| WARNING | Orphan rule / impl move | `IndexType` has an inherent `impl std::fmt::Display` (`crates/axon-schema/src/schema.rs:81-91`) and derives `Hash` (line 71). If `IndexType` moves to `axon-esf` and is re-exported, the `Display` impl **cannot remain in `axon-schema`** (orphan rule violation â€” foreign trait-ish impl on a foreign type is actually a local-type problem: `axon-schema` would be implementing `Display` for a type it no longer owns). The Display impl must move to `axon-esf`. The plan's move list (section 2) and work item 2 never mention moving the impl or preserving the `Hash`/`Eq` derives, which `named_queries.rs` relies on for `IndexType` in match/keying. Two implementers could leave the impl behind and hit a compile error. |
| WARNING | Owned error `message` semantics | The plan defines `SchemaValidationError { instance_path, message, instance }` but never says what `message` contains. The existing classifier (`classify_json_schema_error`, `validation.rs:275`) string-matches `error.to_string()` (e.g. `"x" is a required property`, `is not one of [...]`, `is not of type`) and separately reads `error.instance_path`. If `axon-esf` populates `message` with the **instance-path-prefixed** form, enum extraction via `msg.find('[')` (line 314) can be corrupted by bracketed paths; if it populates the **raw** form, behavior is preserved. This is an unstated invariant where competent implementers will diverge, and the existing enhanced enum/type/required tests are the only guard. |
| WARNING | `default-features = false` and `$ref` resolution | Changing the **workspace** `jsonschema` dep to `default-features = false` removes `resolve-http`/`resolve-file` for `axon-schema` too, not just `axon-esf`. The plan verifies isolation only for `axon-esf` (`cargo tree -p axon-esf`). It does not verify that no existing `entity_schema` or `metadata_schema` relies on external/file `$ref` resolution. If any does, `axon-schema` validation silently breaks. Low likelihood for a self-contained store, but unverified and unmentioned. |
| WARNING | Performance test contradiction | Work item 7 simultaneously requires demonstrating "sub-microsecond" validation **and** "a generous threshold... to avoid noise." These conflict: a hard `< 1Âµs` assertion via `std::time::Instant` is hardware-dependent and CI-flaky (jsonschema `iter_errors` allocates); a generous threshold (e.g. 10Âµs) does not demonstrate sub-microsecond. The plan punts ("if too flaky, add a bench") but leaves the acceptance bar undefined, so the implementer cannot know when item 7 is satisfied. |
| NOTE | Format support under `default-features=false` | The plan asserts formats remain available after disabling default features. This is correct for jsonschema 0.28 (the format-checking deps `email_address`/`idna`/`uuid-simd` are unconditional, and format support is not behind a default feature), and the invalid-`email`/`uuid`/`date-time` tests in item 6 would catch a regression. Self-mitigating, but the plan presents it as fact without the verifying test being called out as the guard. |
| NOTE | Internal/external index shape divergence | `into_collection_schema` splits `IndexDeclaration` into `CollectionSchema.indexes`/`compound_indexes` (kept separate, schema.rs:183/186), while `EsfCoreDocument`/CONTRACT-010 use a unified `indexes:` list. No conversion from internal `CollectionSchema` *back* to the unified `EsfCoreDocument` is defined. External consumers reading a round-tripped schema get the split shape, not the contract shape. Round-trip fidelity is unspecified. |
| NOTE | Indexes path is net-new, low conflict | Confirmed: `EsfDocument` has no `indexes` field today and `into_collection_schema` hardcodes empty vectors (schema.rs:398-399); no other crate parses ESF indexes. So item 5 is purely additive with no competing parser to conflict with â€” lower risk than the plan's caution implies, but it also means **no existing fixture exercises mixed `indexes:`**, so item 6's mixed-declaration test is entirely new ground with no regression baseline. |
| NOTE | Format errors unclassified | Newly-asserted format failures produce a jsonschema error kind with no branch in `classify_json_schema_error`, so they fall to `JsonSchemaErrorKind::Other` with a raw message. Acceptable, but the enhanced-error tests (item 6) don't cover the new error category the change introduces. |

### Verdict: REQUEST_CHANGES

### Summary

The revised plan is architecturally sound and the index-carving work is genuinely additive (ESF indexes are unparsed today, so there is no competing path to break). The blocking-adjacent risks are all in the format-assertion tightening and the type move: the tightening is a workspace-global change applied to two code paths (one beyond the bead's scope), enabled against three existing `date-time` schemas whose data is never audited, and the `Display` impl move plus the undefined `message` field semantics are concrete omissions that will cause either a compile error or silent divergence in the enhanced-error path. Pin down the owned-error `message` contract, move the `IndexType` impls/derives explicitly, scope the format change to what the bead requires (or justify the link-metadata expansion), and audit the three `date-time` sites before relying on the gate.
