### Findings

| Severity | Area | Finding |
|---|---|---|
| BLOCKING | Layer-4 API shape | The plan does not resolve the conflict between current code and ADR-010. Current `CollectionSchema` has separate `indexes: Vec<IndexDef>` and `compound_indexes: Vec<CompoundIndexDef>` ([schema.rs:181]( /Users/erik/Projects/axon/crates/axon-schema/src/schema.rs:181), [schema.rs:185]( /Users/erik/Projects/axon/crates/axon-schema/src/schema.rs:185)), while ADR-010 defines a single ESF `indexes:` list containing both `field` and `fields` forms ([ADR-010:251]( /Users/erik/Projects/axon/docs/helix/02-design/adr/ADR-010-physical-storage-and-secondary-indexes.md:251), [ADR-010:425]( /Users/erik/Projects/axon/docs/helix/02-design/adr/ADR-010-physical-storage-and-secondary-indexes.md:425)). Two implementers could expose incompatible `axon-esf` public shapes. |
| BLOCKING | ESF document boundary | `EsfCoreDocument` / `EntitySchemaDocument` are named but not specified. CONTRACT-010 says an ESF document is YAML or equivalent JSON with `esf_version`, `collection`, optional `description`, `entity_schema`, and `indexes` ([CONTRACT-010:52]( /Users/erik/Projects/axon/docs/helix/02-design/contracts/CONTRACT-010-esf-schema-format.md:52)). The plan also forbids `serde_yaml` in `axon-esf`, so it must explicitly say whether `axon-esf` owns only serde data types or also parse/round-trip document text. |
| BLOCKING | Error contract | The plan says `axon-esf` returns “raw validation errors” and `axon-schema` maps them back, but it does not define an owned raw error shape. Current enhancement depends on `jsonschema::ValidationError` plus the original schema to generate `field_path`, fix suggestions, enum hints, and context ([validation.rs:181]( /Users/erik/Projects/axon/crates/axon-schema/src/validation.rs:181)). Exposing borrowed `jsonschema` errors or only strings will either fail lifetimes or lose structured Axon errors. |
| WARNING | Format assertions | The plan says enable `email`, `uuid`, and `date-time`, but does not require the actual `jsonschema` option. In 0.28.3, `should_validate_formats(true)` is the explicit control ([options.rs:355]( /home/erik/.cargo/registry/src/index.crates.io-1949cf8c6b5b557f/jsonschema-0.28.3/src/options.rs:355)); current Axon validation only sets Draft 2020-12 and builds ([validation.rs:108]( /Users/erik/Projects/axon/crates/axon-schema/src/validation.rs:108)). Relying on draft defaults risks silently not asserting formats. |
| WARNING | Dependency hygiene | `jsonschema` default features enable `resolve-http` and `resolve-file`, and `resolve-http` enables `reqwest` ([jsonschema Cargo.toml:139]( /home/erik/.cargo/registry/src/index.crates.io-1949cf8c6b5b557f/jsonschema-0.28.3/Cargo.toml:139)). The plan says keep workspace `jsonschema` as-is if broader crates rely on it, but current `axon-schema` directly uses workspace `jsonschema` ([axon-schema Cargo.toml:19]( /Users/erik/Projects/axon/crates/axon-schema/Cargo.toml:19)). If `axon-schema` is repointed, that direct dependency should likely be removed or justified. |
| WARNING | Compile-once integration | `validate_entity(&CollectionSchema, &Value)` currently has no place to receive or store a compiled validator and recompiles inside the function ([validation.rs:100]( /Users/erik/Projects/axon/crates/axon-schema/src/validation.rs:100), [validation.rs:108]( /Users/erik/Projects/axon/crates/axon-schema/src/validation.rs:108)). “Change internals to use `CompiledSchema`” is ambiguous unless the plan states that the old API remains per-call while a new API is compile-once, or introduces an explicit cache/compiled wrapper. |
| WARNING | Acceptance coverage | The bead acceptance requires a sub-microsecond compile-once demonstration or bench; the proposed tests list only a compile-once functional test. That leaves a measurable acceptance criterion unplanned. |
| NOTE | Current parser gap | Existing `EsfDocument` does not include `indexes` or `compound_indexes`, and `into_collection_schema` initializes both as empty ([schema.rs:330]( /Users/erik/Projects/axon/crates/axon-schema/src/schema.rs:330), [schema.rs:398]( /Users/erik/Projects/axon/crates/axon-schema/src/schema.rs:398)). Moving Layer-4 types without deciding whether ESF parsing now preserves Layer 4 leaves “preserve behavior” versus “make ESF core useful to pqueue” unresolved. |

### Verdict: REQUEST_CHANGES

### Summary

The plan identifies the right crate boundary, but it leaves the public ESF document shape and Layer-4 index encoding underspecified. The highest rework risk is that `axon-esf` could ship a serde API that preserves current Axon internals while failing the CONTRACT-010/ADR-010 external-consumer shape, or vice versa. The validation wrapper also needs an explicit owned error contract and an explicit format-assertion configuration.
