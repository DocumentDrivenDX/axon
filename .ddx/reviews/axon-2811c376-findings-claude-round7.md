## Findings

I verified the plan against the live tree at `crates/axon-schema/src/schema.rs`, `validation.rs`, `lib.rs`, `crates/axon-storage/src/adapter.rs`, root `Cargo.toml`, and `Cargo.lock`.

| Severity | Area | Finding |
|---|---|---|
| BLOCKING | Moved `IndexType` definition | The plan's verbatim `IndexType` is `enum { Hash, BTree, FullText }`, and it claims moved types "preserve their existing serde shape and public derives" and that `IndexType` "keeps its lowercase Display implementation." This is factually false. The actual enum (`schema.rs:71-91`) is `enum { String, Integer, Float, Datetime, Boolean }` with a `Display` that maps those five variants. An implementer copying the plan literally inverts the type. This breaks the workspace: `axon-storage/src/adapter.rs:98-109` exhaustively matches `String/Integer/Float/Boolean/Datetime` to route `IndexValue` (would fail to compile against `Hash/BTree/FullText`), plus ~12 call sites (`axon-cypher/src/storage_adapter_store.rs:416-432`, `axon-server/src/gateway.rs:6002`, `axon-cypher/tests/sqlite_parity.rs`, `axon-api/src/test_fixtures.rs`, `axon-graphql`, `axon-mcp`) construct those value-type variants, and every existing ESF/index doc and test uses `type: string\|integer\|...`. `Hash/BTree/FullText` are index *algorithms*, not the *value types* this enum encodes; the plan conflated the two. The moved `IndexType` must be `String/Integer/Float/Datetime/Boolean` with its existing `Display`. As written the spec is self-contradictory and not executable. |
| WARNING | `validate_entity`/`compile_entity_schema` rewiring | The plan requires `validate_entity` and `compile_entity_schema` to consume `axon-esf` types (`SchemaCompileError.message`, `RawValidationError`), but does not state that the existing `enhance_json_schema_error`/`classify_json_schema_error` (`validation.rs:182-309`) must be re-typed from `&jsonschema::ValidationError` to `&RawValidationError`. Those functions call `error.instance_path`, `error.instance.as_ref()` (a `Cow`), and `error.to_string()`; the new struct exposes `instance_path: String`, `instance: Value`, `message: String`. The mechanical edits (`.as_ref()` removal, `raw_msg` ← `message`) are derivable but unspecified, leaving the classifier's exact rewiring to implementer judgment. |
| NOTE | Format-validation vs `default-features = false` | Verified sound: `Cargo.lock:2302-2323` shows `email_address`, `idna`, `uuid-simd`, `fraction` are unconditional `jsonschema` deps while `reqwest` arrives via the (default) resolve features. So disabling default features keeps format assertion intact and only drops resolve — the plan's "rejects malformed email/uuid/date-time" claim holds. Blast radius is contained: `axon-schema` is the only crate depending on `jsonschema`, and no external `$ref`/resolver usage exists in `axon-schema` or `axon-registry`. |
| NOTE | Format asymmetry | `should_validate_formats(true)` is specified only for `axon-esf::CompiledSchema::compile`. `validate_link_metadata` stays on the direct `jsonschema` path (`validation.rs:139-164`) without it, so link-metadata format strings remain unchecked while entity formats become asserted. Intentional per the plan's scoping, but worth an explicit test note so the divergence isn't read as a bug later. |
| NOTE | Stray `schema.rs.tmp` | `crates/axon-schema/src/schema.rs.tmp` (0 bytes) exists in the tree — residue from a prior edit. Harmless (not a module), but should be removed during the change to keep `git status` clean. |

### Verdict: REQUEST_CHANGES

### Summary

The plan is converged on the `axon-esf` extraction, draft/format/error semantics, document carriers, and `EsfDocument` integration — all checks against the real code passed. It is blocked by one concrete, literal-spec error: the moved `IndexType` is given as `Hash/BTree/FullText` while the actual enum is `String/Integer/Float/Datetime/Boolean`, and the plan asserts the move "preserves the existing serde shape." Implementing it verbatim breaks the storage/cypher/server/graphql/mcp layers and all index tests. Fix the `IndexType` variants (and its `Display`) to match the existing enum and specify the classifier's re-typing to `RawValidationError`; the rest is ready.
