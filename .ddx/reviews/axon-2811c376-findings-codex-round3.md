### Findings

| Severity | Area | Finding |
|---|---|---|
| BLOCKING | Public `axon-esf` API | The plan says to add `IndexDeclaration`, `EntitySchemaDocument`, and `EsfCoreDocument`, but does not define their exact fields, derives, module exports, parse helpers, or serialization contract. This matters because current index structs are `Serialize + Deserialize` ([schema.rs:93](/Users/erik/Projects/axon/crates/axon-schema/src/schema.rs:93), [schema.rs:123](/Users/erik/Projects/axon/crates/axon-schema/src/schema.rs:123)), while the plan only specifies custom `Deserialize` for `IndexDeclaration`. Two implementers could produce incompatible public APIs, especially for serializing mixed `indexes:` back to ESF. |
| BLOCKING | Leaf dependency boundary | “No forbidden deps in tree” is not testable because the forbidden dependency set is never defined. The target is a leaf crate for external consumers, but current `CollectionSchema` depends on `axon_core::CollectionId` ([schema.rs:390](/Users/erik/Projects/axon/crates/axon-schema/src/schema.rs:390)), and an implementer could accidentally pull `axon-core`, `axon-schema`, HTTP, or storage-side crates into `axon-esf` without violating a written rule. |
| WARNING | Enhanced validation errors | The plan says `axon-esf` exposes owned raw errors and `axon-schema` enhanced errors remain unchanged, but it does not explicitly require refactoring the enhancer to consume `{ instance_path, message, instance }`. Today the enhancer takes `jsonschema::ValidationError` directly ([validation.rs:181](/Users/erik/Projects/axon/crates/axon-schema/src/validation.rs:181)) and uses `error.instance` for enum/type classification ([validation.rs:290](/Users/erik/Projects/axon/crates/axon-schema/src/validation.rs:290), [validation.rs:301](/Users/erik/Projects/axon/crates/axon-schema/src/validation.rs:301)). This is likely implementable, but it is still an undocumented adapter boundary. |
| WARNING | ESF index compatibility | Current `EsfDocument` has no `description`, `indexes`, or `compound_indexes` fields ([schema.rs:330](/Users/erik/Projects/axon/crates/axon-schema/src/schema.rs:330)), and conversion currently sets both index vectors empty ([schema.rs:397](/Users/erik/Projects/axon/crates/axon-schema/src/schema.rs:397)). The plan defines new mixed `indexes:` behavior but does not say whether any split `compound_indexes:` ESF input should be rejected, ignored, or also accepted. |
| WARNING | Performance evidence | The release example still bakes in an absolute `< 1 microsecond` failure threshold. Moving it out of debug tests helps, but `cargo run -p axon-esf --release --example compiled_schema_perf` can still fail due to host variance rather than implementation regression, making closure evidence environment-sensitive. |
| NOTE | `$ref` dependency change | The dependency decision is coherent: workspace `jsonschema` is currently default-featured ([Cargo.toml:94](/Users/erik/Projects/axon/Cargo.toml:94)), and `jsonschema` defaults enable `resolve-http` and `resolve-file` ([Cargo.toml:139](/home/erik/.cargo/registry/src/index.crates.io-1949cf8c6b5b557f/jsonschema-0.28.3/Cargo.toml:139)). The plan should still name internal `#/$defs` `$ref` support as retained and tested, because “external/file `$ref` is out” can otherwise be over-implemented as “all `$ref` is out.” |

### Verdict: REQUEST_CHANGES

### Summary

The plan is close, but not yet converged enough for low-variance implementation. The biggest remaining gaps are the exact public `axon-esf` API/serde contract and the dependency boundary that defines what “leaf crate” means. The validation and `$ref` decisions are mostly workable, but they need sharper adapter and test wording to prevent competent implementers from making incompatible choices.
