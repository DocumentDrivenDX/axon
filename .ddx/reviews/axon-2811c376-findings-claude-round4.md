I now have enough grounded evidence. Key verifications:
- `should_validate_formats` appears **nowhere** in the current codebase — entity validation does *not* validate formats today.
- jsonschema 0.28.3 compiles `email_address`, `uuid-simd`, `fancy-regex`, `idna` **unconditionally** (Cargo.lock:2306–2324); only `reqwest` is behind the `resolve-http` default feature. So `default-features = false` drops reqwest/hyper/tower while keeping format validators.
- `EsfDocument` has **no** existing `indexes`/`description`/`compound_indexes` fields — additions are purely additive.
- Only `axon-schema` depends on `jsonschema` workspace-wide.

## Findings

| Severity | Area | Finding |
|---|---|---|
| BLOCKING | Validation semantics | The plan silently turns on JSON-Schema **format assertion** for entity validation. Today `validate_entity` (`validation.rs:108–110`) and `compile_entity_schema` (`validation.rs:171–176`) build with `.with_draft(Draft202012).build()` and **never** call `should_validate_formats` — it appears nowhere in the repo. The plan routes both through `CompiledSchema::compile`, which *mandates* `should_validate_formats(true)`. Because `email_address`/`uuid-simd`/`fancy-regex`/`idna` are unconditional jsonschema deps (Cargo.lock:2310–2323), formats become *live* even with `default-features=false`. Net effect: any `entity_schema` declaring `format:` whose data violates it — currently accepted — will now be **rejected** on the write path. The plan frames this as "Refactor entity validation only" and the test list claims "existing enhanced required/enum/type tests pass unchanged," neither acknowledging nor testing the change. Two competent implementers diverge here: one preserves today's no-format behavior; one follows the literal `CompiledSchema` spec. The plan must explicitly decide formats on/off for the entity path. |
| WARNING | External API / `EsfCoreDocument` | `EsfCoreDocument` carries `indexes` but **no `compound_indexes`**, while `axon-schema::EsfDocument` gains a legacy `compound_indexes` field "for compatibility with current split Axon schema JSON shapes." No `deny_unknown_fields` is specified, so an external ESF doc using the split shape fed to `EsfCoreDocument` will **silently drop** its `compound_indexes:` entries. The plan addresses legacy-compound compatibility only for `EsfDocument`, not for the public `EsfCoreDocument` that pqueue consumes — a silent data-loss gap for exactly the external consumer this crate exists to serve. |
| WARNING | Error types | `SchemaCompileError` derives only `Debug, Clone, PartialEq, Eq` — no `Display`, no `std::error::Error`. But the preserved-prefix instructions are written as `invalid schema definition: {raw}` / `invalid schema: {raw}`. As literally written, `format!("...{raw}")` won't compile; the implementer must reach into `raw.message` or add a `Display` impl. For a *public* leaf crate, shipping error types with no `Display`/`Error` impls is itself a likely-rework interface decision the plan leaves unspecified (same for `RawValidationError`/`RawValidationErrors`). |
| NOTE | Error parity | The compile-failure path currently emits `SchemaValidationError::new("/", …)` (field_path `"/"`). The plan preserves the message *prefix* but never states the `"/"` field_path is preserved. An implementer optimizing the refactor could change it. |
| NOTE | `IndexDeclaration` deser | Disambiguation (field/fields/both/neither) is specified, but unknown-field strictness (`deny_unknown_fields`) and the empty-object case are not. Combined with `#[serde(untagged)] Serialize` plus a hand-written `Deserialize`, strictness is left to implementer judgment — two implementers could differ on whether `{}` or `{field, extra}` is rejected. |
| NOTE | Re-export surface | `IndexDeclaration` is new and reachable through `EsfDocument.indexes`/`EntitySchemaDocument.indexes`, but the plan mandates its re-export only in `schema.rs`, not in `lib.rs` (`lib.rs:46–50` re-exports the sibling index types at crate root). `axon_schema::IndexDeclaration` would not resolve, unlike `axon_schema::IndexDef`. Minor asymmetry worth a deliberate decision. |
| NOTE | Dep intent | `validate_link_metadata` keeps a direct `jsonschema` call, so `axon-schema` *retains* `jsonschema` — correct, but the plan's step 1 ("Add axon-esf") never states this explicitly; a reader could infer the goal was to remove jsonschema from axon-schema. Leaves two divergent error-shaping paths (esf `RawValidationError` vs. inline string-join) — acceptable for this bead but an undocumented inconsistency. |

## Verdict: REQUEST_CHANGES

## Summary

The dependency boundary, type moves, and `default-features=false` strategy are well-specified and verifiably sound (format validators survive; only reqwest/hyper/tower drop; EsfDocument additions are non-breaking). The blocker is unacknowledged: routing entity validation through `CompiledSchema::compile`'s mandatory `should_validate_formats(true)` silently changes the production write-path validation contract, since today's `validate_entity` does no format checking. Combined with the `EsfCoreDocument` silent-drop of legacy `compound_indexes` and the no-`Display` error types, the plan needs targeted clarification — not redesign — before it has converged enough to implement.
