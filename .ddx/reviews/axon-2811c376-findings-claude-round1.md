### Findings

| Severity | Area | Finding |
|---|---|---|
| BLOCKING | Dependency isolation / acceptance test | The plan keeps `axon-schema`'s `jsonschema.workspace` "as-is" (default features → `resolve-http` → `reqwest`, confirmed in `Cargo.lock:2320`) while `axon-schema` now depends on `axon-esf`. With `resolver = "2"` (root `Cargo.toml:18`), Cargo unifies normal-dependency features across the workspace into a single `jsonschema` node. `axon-schema` is the **only** crate using `jsonschema` (`grep` of all `crates/*/Cargo.toml` returns only `axon-schema`), so its default features stay enabled, `resolve-http` stays on the shared node, and `cargo tree -p axon-esf` traverses `axon-esf → jsonschema → reqwest`. The bead's own acceptance check `cargo tree -p axon-esf \| rg 'reqwest\|hyper\|tower'` will therefore **fail**. The plan's hedge ("keep workspace jsonschema as-is if broader Axon crates still rely on it") rests on a false premise — nothing else relies on it, so `axon-schema` must also move to `default-features = false`, which the plan does not commit to. |
| BLOCKING | Behavior preservation vs. format assertions | The bead requires "Preserve existing Axon behavior" **and** "Enable format assertions for `email`, `uuid`, and `date-time`." These conflict. Current `validate_entity` (`validation.rs:108-116`) builds the validator **without** `should_validate_formats`, so per Draft 2020-12 `format` is annotation-only and malformed values pass. Fixtures already carry `"format": "date-time"` (`named_queries.rs:568`). Turning on format assertion newly **rejects** previously-accepted entities — a behavior change, not a preservation. The plan never acknowledges this regression risk nor scopes which entities/tests it affects. |
| BLOCKING | jsonschema format API granularity | `jsonschema` 0.28 has no per-format toggle: the only switch is `should_validate_formats(true)`, which asserts **all** built-in formats, not just the three named (`email`/`uuid`/`date-time`). "Enable format assertions for email, uuid, and date-time" is thus unimplementable as literally written with the stock API — two implementers will diverge (global on vs. custom-registering only three). The plan gives no resolution. Additionally, `default-features = false` must be verified not to drop the `email_address`/`uuid-simd`/`idna` format backends (present in `Cargo.lock:2310-2323`); the plan asserts only that it drops HTTP resolution and does not name which features to re-enable to keep the three required validators live. |
| BLOCKING | CompiledSchema raw-error interface underspecified | The plan says `axon-esf` returns "raw validation errors" and `axon-schema` maps them into the enhanced `SchemaValidationError`. But the existing enhancement (`validation.rs:182-309`) reads `error.instance` (the actual JSON `Value`, used by `format_value_brief`/`json_type_name` for enum/type "actual" reporting) and `error.instance_path`, not just the message string. If the raw-error type carries only `{path, message}`, the tests `enum_mismatch_includes_allowed_values_and_actual`, `type_mismatch_shows_expected_and_actual`, and the `did-you-mean` path degrade or break. The raw-error contract must carry `instance_path` + the instance `Value` + message, and the plan does not specify this — guaranteeing rework or a silent regression. |
| WARNING | Public API: error ownership/lifetime | `jsonschema` 0.28 `ValidationError<'a>` borrows from the instance/schema. A `CompiledSchema::validate(&self, data)` that returns errors out of the call must produce **owned** errors (`to_owned()` / a bespoke owned struct). The plan does not address ownership, and the natural-but-wrong design (returning borrowed `ValidationError`) will not compile across the `axon-esf → axon-schema` boundary. |
| WARNING | EsfCoreDocument vs. existing EsfDocument | The plan keeps `EsfDocument` (with `parse`, `into_collection_schema`, all eight layers) in `axon-schema` and adds `EsfCoreDocument`/`EntitySchemaDocument` in `axon-esf`, but never defines their relationship or serde shapes. Two doc types deserializing overlapping ESF JSON invites drift. Also, `axon-esf` is restricted to `serde`/`serde_json`/`jsonschema` only — so it **cannot parse YAML** (current ESF is YAML-first, `schema.rs:355` via `serde_yaml`). External consumers (pqueue/Cayce) loading ESF through `axon-esf` get JSON-only ingestion, which may not match how ESF is authored. Unstated constraint. |
| WARNING | Partial repoint leaves inconsistent validation | The plan only repoints `validate_entity` to `axon-esf::CompiledSchema`. `validate_link_metadata` (`validation.rs:139-143`) and `compile_entity_schema` (`validation.rs:171-176`) keep calling `jsonschema` directly. Result: entity validation asserts formats while link-metadata validation does not, and `axon-schema` retains a direct `jsonschema` dependency anyway (so the isolation benefit is local to `axon-esf` only). Either move all three or the rationale for splitting them needs stating. |
| NOTE | Workspace-dependency convention | CLAUDE.md says shared deps go in `[workspace.dependencies]`. `axon-esf` must use a **direct** `jsonschema = { version = "0.28", default-features = false }` because Cargo rejects `default-features = false` on an inherited workspace dep. This is justified but is a deviation worth calling out; it also means two `jsonschema` declarations (workspace + direct) at the same version, which is exactly what drives the unification problem in the first finding. |
| NOTE | Thin holder field set ambiguous | The bead's "Layer-4 index declarations" spans both `indexes: Vec<IndexDef>` **and** `compound_indexes: Vec<CompoundIndexDef>` (`schema.rs:181-186`). The plan lists the four index types plus `EsfCoreDocument` but never states which fields the holder carries, so an implementer could omit `compound_indexes`. |

### Verdict: REQUEST_CHANGES

### Summary

The crate carve-out boundary (move index types + a holder, keep `CollectionId`-bound `CollectionSchema` and the enhanced errors in `axon-schema`) is sound, but the plan will not pass its own acceptance criteria as written. Feature unification under `resolver = "2"` means `cargo tree -p axon-esf` still shows `reqwest` unless `axon-schema` *also* drops `jsonschema` default features — which the plan explicitly declines to do on a false premise. Separately, "enable format assertions" both contradicts "preserve existing behavior" and is not expressible as a three-format subset in `jsonschema` 0.28's API, and the "raw validation errors" interface is underspecified in a way that risks regressing the structured-error tests. Resolve the unification/default-features decision, the format-assertion scope-and-behavior conflict, and the raw-error type contract before implementing.
