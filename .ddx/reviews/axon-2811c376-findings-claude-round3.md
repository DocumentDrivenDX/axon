### Findings

| Severity | Area | Finding |
|---|---|---|
| BLOCKING | New document types | Step 5 adds `EntitySchemaDocument` and `EsfCoreDocument` to `axon-esf`, but **neither type is defined anywhere in the plan** — no fields, no purpose, no relationship to `axon-schema::EsfDocument`. Every prose decision under "Fixed Decisions" discusses `axon-schema::EsfDocument` (description, mixed `indexes`); these two new `axon-esf` types appear only as bare names in one step. Two implementers would invent entirely different types (is `EsfCoreDocument` a leaf-crate replacement for `EsfDocument`? a subset? a parallel parser?). Not converged. |
| BLOCKING | Public name collision | The plan's `axon-esf::SchemaValidationError { instance_path, message, instance }` and `SchemaValidationErrors(pub Vec<_>)` collide by name **and tuple shape** with the existing public `axon-schema` types: `SchemaValidationError { field_path, message, severity, fix, context }` (`validation.rs:15`) and `SchemaValidationErrors(pub Vec<_>)` (`validation.rs:61`), which step 7 says "remain unchanged." Two different public structs with identical names will coexist and `axon-schema` must import the `axon-esf` one. The plan never acknowledges the collision nor specifies disambiguation (rename, alias, or path-qualify). |
| BLOCKING | CompiledSchema interface + classifier feeding | The plan defines `CompiledSchema::compile` but **never defines the method that validates data or what it returns**. Meanwhile `axon-schema::validate_entity` must keep producing enhanced errors via `enhance_json_schema_error`/`classify_json_schema_error` (`validation.rs:182,275`), which currently consume `&jsonschema::ValidationError` and read `.instance` (a `Value`), `.instance_path`, and `.to_string()`. Because "no public API returns `jsonschema` internals" and `axon-esf`/`axon-schema` are separate crates (`pub(crate)` cannot cross), `axon-schema` can only receive `axon-esf`'s owned errors and **must refactor both classifier functions to consume the owned `instance_path/message/instance` fields**. This refactor is not listed as a step, and the validate-method signature is undefined. |
| WARNING | Compile-error message format | `SchemaCompileError.message` content is unspecified. Today `compile_entity_schema` returns `AxonError::SchemaValidation("invalid schema: {e}")` (`validation.rs:176`) and `validate_entity`'s build-failure path returns `"invalid schema definition: {e}"` (`validation.rs:114`). If both now wrap `SchemaCompileError.message`, whether `message` is the raw `{e}` or already-prefixed determines double-prefixing / changed strings. Existing tests assert these prefixes (e.g. `validation.rs` build-error tests). |
| WARNING | IndexDeclaration Serialize | Only the custom `Deserialize` is specified. The moved types derive `Serialize` and are round-tripped in tests (e.g. `serde_json::to_string` of an ESF doc, `schema.rs:706`). Whether `IndexDeclaration` needs `Serialize`, and how a custom-`Deserialize`/derived-`Serialize` asymmetry round-trips, is undefined. |
| WARNING | EsfDocument "existing" indexes | `EsfDocument` (`schema.rs:330`) has **no** `indexes`/`compound_indexes` fields today; `into_collection_schema` hardcodes them empty (`schema.rs:398-399`). Step 8's "split mixed declarations into **existing** `indexes` and `compound_indexes`" is misleading — these are net-new fields on `EsfDocument` plus new wiring in `into_collection_schema`. A new `Vec<IndexDeclaration>` field also needs `#[serde(default)]` (current `Option<Value>` fields rely on serde's Option special-case; a `Vec` does not). |
| NOTE | Format-assertion audit scope | The audit enumerated only `axon-schema`/one `axon-api` site, but actual JSON-Schema `format` assertions in validation paths are narrow: `handler.rs:16704` and `fixtures/beads.esf.yaml:53` (both audited). Exposure is largely covered; the gap is that cross-crate test data (`axon-api` tests, `axon-graphql`) was not explicitly enumerated, though the "full test suite catches regressions" mitigation is sound. |
| NOTE | Eq-omission rationale is wrong | "Do not derive `Eq` for `IndexDeclaration`… to avoid a compile-time contradiction." All fields of `IndexDef`/`CompoundIndexField`/`CompoundIndexDef` are `Eq`-capable (`String`, `bool`, and `IndexType` which already derives `Eq`, `schema.rs:71`) — deriving `Eq` would compile. The decision (PartialEq-only, for consistency) is fine, but the stated justification is factually incorrect and may mislead the implementer. |
| NOTE | Owned `instance: Value` cost | Each owned `SchemaValidationError` carries a full clone of the instance `Value`; the enhanced classifier only needs a brief value (`format_value_brief`/`json_type_name`). The perf example uses "small records" and won't surface per-error instance-cloning cost on large entities. |
| NOTE | default-features=false is safe (verified) | Confirmed only `axon-schema` depends on `jsonschema` and no resolver/`referencing` APIs are used (grep clean), so the workspace-wide `default-features = false` is low-risk for compilation/`$ref`. Format validators in `jsonschema` 0.28 are not gated by the default `resolve-http`/`resolve-file` features, so `should_validate_formats(true)` keeps working; if that assumption were wrong, the step-9 invalid-format tests fail loudly rather than silently. The plan never states an explicit check that formats survive feature-stripping — worth one line. |

### Verdict: REQUEST_CHANGES

### Summary

The design direction is sound and most round-2 decisions are concrete, but the plan has not converged enough to hand to an implementer. Three convergence gaps are blocking: two brand-new public types (`EntitySchemaDocument`, `EsfCoreDocument`) are named but never defined; the new `axon-esf::SchemaValidationError(s)` collides by name and shape with the existing `axon-schema` types that step 7 keeps unchanged; and the central `CompiledSchema` validation method and the required refactor of `enhance/classify_json_schema_error` (which today read `jsonschema::ValidationError` internals) are unspecified. Resolve the type definitions, the naming collision, and the error-classifier interface, and this is ready.
