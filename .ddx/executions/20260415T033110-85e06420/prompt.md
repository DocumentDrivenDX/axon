    <title>auth-federation: users/user_identities auto-provision + Tailscale synthesis</title>
    <description>
Add the federation layer that takes an external identity (Tailscale tailnet handle today, OIDC/email tomorrow) and resolves it to a first-class axon User with an idempotent auto-provision. This is the non-JWT arm of the auth pipeline. The JWT arm is done (B1). The middleware wiring that dispatches between the two arms depends on D1 (path router) — this bead stays focused on the federation primitive and its storage/test surface.&#xA;&#xA;## What is in this bead&#xA;&#xA;1. Storage adapter extension: one new trait method that atomically upserts a user_identities row and its parent users row, returning the User. Implemented on all three backends (memory, sqlite, postgres) using the ON CONFLICT DO NOTHING pattern from ADR-018 §6.&#xA;&#xA;2. A federation helper module in axon-server that takes a Tailscale whois response (reuse TailscaleWhoisResponse from existing crates/axon-server/src/auth.rs) and resolves it to a User via the new storage method.&#xA;&#xA;3. Tests proving the 64-way concurrent race converges on one users row, and that adding a second provider for the same human keeps the users row stable while adding a user_identities row.&#xA;&#xA;## What is NOT in this bead (defer to follow-up beads)&#xA;&#xA;- Middleware wiring of the Tailscale path into verify_request — depends on D1 (path router) to know which tenant the URL names; that&#39;s a later bead.&#xA;- PROP-011 Tailscale↔JWT equivalence property test — same reason; requires full middleware.
