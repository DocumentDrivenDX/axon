Finish wiring the ADR-018 auth stack into the real request pipeline. After the smoke test on commit e83cf02, D3 already landed more than expected: `resolve_tenant_handler` composes `{tenant}:{database}` into the TenantRouter slug, giving real per-pair physical isolation. What remains is the three things that are *built but not installed*.&#xA;&#xA;## What is already live (confirmed by smoke test)&#xA;&#xA;- Path-based routing: /tenants/:t/databases/:d/... routed to per-pair SQLite files via composite slug&#xA;- GraphQL under the nested path&#xA;- Unprefixed REST + /graphql return 404&#xA;- HttpAxonClient in the SDK + the UI use the new paths&#xA;- Legacy X-Axon-Database header is gone from REST&#xA;&#xA;## What is still NOT installed (the gap this bead closes)&#xA;&#xA;### 1. JWT pipeline not enforced&#xA;&#xA;B1 landed `auth_pipeline::jwt_verify_layer` + `JwtIssuer`. The real gateway router in `crates/axon-server/src/gateway.rs` does NOT install this layer. The server currently runs in legacy auth mode (Tailscale or --no-auth). Install the layer as an OPTIONAL middleware: if the request has an Authorization header, run the JWT pipeline; if not, fall back to legacy auth.&#xA;&#xA;Wire point: in `build_router_with_auth` (or wherever the data-plane nested router is constructed), add `.layer(from_fn_with_state(state.clone(), jwt_verify_layer))` before the tenant_handler middleware. The layer needs access to a `JwtIssuer` and the `StorageAdapter`. Add them to the AppState struct.&#xA;&#xA;Create the JwtIssuer at server startup in `serve.rs`. Read the signing key from an env var `AXON_JWT_KEY` (base64 HMAC secret); if unset, generate a random 32-byte key at startup and log a warning (dev mode). Production key management is a future concern.&#xA;&#xA;The legacy auth path (crates/axon-server/src/auth.rs Identity + Role) still works for requests without an Authorization header — the JWT layer is a no-op pass-through when the header is absent. This preserves --no-auth, --guest-role, and Tailscale whois behavior exactly as today.&#xA;&#xA;### 2. Audit attribution not populated&#xA;&#xA;F1 added `AuditAttribution` as an optional field on `AuditEntry` plus the `with_attribution()` builder. No handler currently calls it. In the JWT middleware (step 1), after `ResolvedIdentity` is installed into the request extension, also expose a `current_attribution()` helper on AppState that the audit write sites can call.&#xA;&#xA;Find every `AuditEntry::new(...)` call site in `crates/axon-server/src/gateway.rs` (there are probably 10-20) and chain `.with_attribution(current_attribution(&amp;request))` when a ResolvedIdentity is present. When absent (legacy auth), leave the attribution as None — the existing `actor: String` field still works.&#xA;&#xA;### 3. gRPC service.rs still reads x-axon-database&#xA;&#xA;`crates/axon-server/src/service.rs:26` has `const AXON_DATABASE_HEADER: &amp;str = &#34;x-axon-database&#34;` and `grpc_current_database` reads it from metadata. The gRPC service is entirely separate from the REST gateway and D3 did not touch it. Replace this header read with path-based extraction... except gRPC doesn&#39;t have URL paths in the same way. gRPC calls use service + method names, not paths. The database has to come from either:&#xA;&#xA;(a) A metadata field (still a header-equivalent), OR&#xA;(b) A new field on every request proto&#xA;&#xA;(a) is pragmatic. Rename the metadata header from `x-axon-database` to `x-axon-tenant-database` and carry a composite value `&#34;{tenant}:{database}&#34;`. This matches the REST composite slug. The gRPC service parses it and dispatches via the same `get_or_create_any` path.&#xA;&#xA;Also add `x-axon-authorization` (or reuse the standard `authorization` metadata key) for JWT credentials. Run the JWT verify pipeline in the gRPC `check_auth` step before dispatching to the handler.&#xA;&#xA;For this bead, the gRPC change is small:&#xA;- Rename the metadata constant&#xA;- Update `grpc_current_database` to parse the composite form&#xA;- Update the Rust + TS gRPC clients to send the composite form&#xA;- Add JWT auth path in `check_auth`&#xA;&#xA;### 4. DatabaseRouter NOT swapped in (deferred — not needed right now)&#xA;&#xA;D2 landed `DatabaseRouter` as a new primitive, but the gateway still uses `TenantRouter::get_or_create_any(slug)`. The smoke test confirmed this composite slug works correctly. Replacing TenantRouter with DatabaseRouter is a refactor that doesn&#39;t change behavior — defer it to a later cleanup bead. Not in this bead&#39;s scope.&#xA;&#xA;## Scope summary&#xA;&#xA;This bead does exactly three things:&#xA;&#xA;1. Migrate gRPC service.rs from x-axon-database header to the composite slug + optional JWT metadata&#xA;&#xA;## Tests for this bead&#xA;&#xA;### Integration test: crates/axon-server/tests/cutover_jwt_test.rs (NEW)&#xA;&#xA;- create a JwtIssuer with a known key&#xA;- mint a JWT for a default user with grants on (default, default)&#xA;- start the in-process test server harness that gateway_test.rs uses&#xA;- send a request with Authorization: Bearer &lt;jwt&gt; to /tenants/default/databases/default/collections&#xA;- assert 200 (the JWT pipeline accepted + the handler saw ResolvedIdentity)&#xA;&#xA;### Integration test: attribution populated&#xA;&#xA;- send an entity-create request with a JWT&#xA;- query the audit log&#xA;- assert the audit entry has `attribution.user_id` set to the JWT sub&#xA;&#xA;### gRPC test: composite slug&#xA;&#xA;- send a CreateEntity gRPC request with metadata `x-axon-tenant-database: acme:orders`&#xA;- assert routing to the acme:orders database (check the storage file exists)&#xA;&#xA;### Manual smoke test&#xA;&#xA;Start the server, issue a JWT via `/control/tenants/default/credentials`, use the HttpAxonClient with the JWT, create an entity, verify the audit log has attribution.
