## 3. Intent Summary

- Vision: Axon is an entity-first OLTP store that is schema-driven, auditable by default, and agent-accessible across JSON, GraphQL, and MCP (`product-vision.md:10-23`).
- Requirements: P0 requires audit, CRUD, transactions, API, embedded mode, and CLI; P1 adds schema evolution, aggregation, server mode, auth, admin UI, multi-tenancy, GraphQL, MCP, control plane, and related operator surfaces (`prd.md:280-308`).
- Architecture: Server mode must be stateless and delegate durability to a selected backing store; SQLite is the embedded baseline and PostgreSQL is the required server backend (`technical-requirements.md:23-42`, `technical-requirements.md:60-69`).
- Concerns: Active repo concerns are `rust-cargo` and `svelte-bun`, with rust-cargo explicitly requiring strict clippy, `cargo deny`, `cargo machete`, and workspace tests, while the UI concern is intended to govern a Bun-based SvelteKit toolchain (`concerns.md:3-34`).
- Test plan: The test suite is meant to define truth, with backend parity and API contract coverage in CI and ratchets for correctness and audit gaps (`test-plan.md:19-39`, `test-plan.md:52-62`).
- Implementation plan state: Core embedded functionality is present and testable, but several P1/P2 operator and deployment surfaces remain scaffolded, stubbed, or undocumented in the shipped binary.

## 5. Implementation Map

- Topology: Cargo workspace members are `axon-core`, `axon-schema`, `axon-audit`, `axon-storage`, `axon-api`, `axon-cli`, `axon-sim`, `axon-server`, `axon-graphql`, `axon-mcp`, and `axon-render`.
- Entry points: `crates/axon-cli/src/main.rs` is the embedded operator surface; `crates/axon-server/src/main.rs` is the HTTP, gRPC, and MCP stdio server entry point.
- Public interfaces: HTTP and gRPC are expected to front the same handler surface, while MCP layers agent tools on top of handler logic; UI is expected to call the HTTP gateway and be served by `axon-server`.
- Test surfaces: Workspace tests currently pass end to end; strict clippy fails; CI currently covers check, test, basic clippy, and fmt only; backend parity for PostgreSQL is not integrated into the same always-on quality path as SQLite.
- Unplanned or weakly planned areas: server runtime is hardwired to `MemoryStorageAdapter`, the admin UI is scaffold-only, several MCP tools are placeholders, and active beads lack the context-digest hygiene required by the HELIX workflow.
