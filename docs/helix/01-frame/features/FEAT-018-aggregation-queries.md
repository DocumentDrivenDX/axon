# Feature Specification: FEAT-018 — Aggregation Queries

**Feature ID**: FEAT-018
**Status**: draft
**Priority**: P1
**Owner**: Core Team
**Requirement Prefix**: AGG
**Covered PRD Subsystem(s)**: Entity-Graph Data Model
**Covered PRD Requirements**: FR-3 (aggregation facet of the policy-aware read model; traversal and pattern facets are owned by FEAT-009)
**Cross-Subsystem Rationale**: None — single subsystem. The GraphQL/MCP/SDK exposures are projections generated by FEAT-015/FEAT-016 surface generation, not separate capabilities of this feature.

## Overview

Aggregation queries compute summary statistics over entities in a collection
— count, sum, average, minimum, maximum — with optional grouping and
filtering. This feature implements the aggregation facet of PRD FR-3 (one
policy-aware read model covering filtering, sorting, pagination, aggregation,
and graph traversal).

Per product-owner decision (2026-06-10), aggregations are **projections of
the unified Cypher read planner**
([FEAT-009 — Unified Graph Query (Cypher)](FEAT-009-unified-graph-query.md),
ADR-021,
[CONTRACT-007](../../02-design/contracts/CONTRACT-007-cypher-query-surface.md))
— not a standalone query engine. Every aggregation is expressible in the V1
Cypher subset and compiles through the same plan, limits, and policy
enforcement as every other read. Clients reach aggregations through the
structured API, GraphQL, and MCP projections of that one planner.

## Ideal Future State

A dashboard builder or agent asks "how many invoices per status?" or "what is
the average priority of in-progress tasks?" and gets an answer in one
request, through whichever surface they already use — a generated GraphQL
aggregation field, a generated MCP aggregation tool, the structured API, or
an ad-hoc Cypher query — and every one of those answers is computed by the
same planner, accelerated by the same indexes, bounded by the same limits,
and filtered by the same policy decisions. Hidden rows never contribute to
totals, and a malformed aggregation fails with the same structured error
everywhere.

## Problem Statement

- **Current situation**: Agents and dashboards need summary data ("how many
  beads are blocked?", "count beads by status"); without server-side
  aggregation, clients fetch all entities and compute summaries client-side.
- **Pain points**: Client-side summarization is wasteful and slow for large
  collections, and any hand-built aggregation path risks diverging from the
  read model's policy filtering, leaking hidden rows through counts.
- **Desired outcome**: Server-computed, policy-safe aggregations with
  grouping, available with identical semantics through every read surface,
  bounded in memory by group cardinality rather than entity count.

## Functional Areas

| Area | User question or job | Feature responsibility |
|------|----------------------|------------------------|
| Aggregation semantics | "What summaries can I compute, over which fields?" | Functions, grouping, null handling, type rules — as planner projections |
| Surface projections | "How do I call aggregations from my client?" | Structured API, GraphQL, and MCP projections with parity |

## Requirements

### Functional Requirements by Area

#### Aggregation Semantics

- **AGG-01**: Aggregations must support count, sum, average, minimum, and
  maximum over collection entities, combinable with the read model's
  filters. Count applies to all field types; sum/average to numeric fields;
  minimum/maximum to numeric, string, and datetime fields. A single request
  may compute multiple aggregations over different fields.
- **AGG-02**: Results must support grouping by one or more fields, returning
  each group's key value(s) alongside its aggregated results. Entities where
  a grouped field is null or missing form their own null-labeled group.
- **AGG-03**: Null or missing values in an aggregated field must be excluded
  from sum/average/minimum/maximum (not treated as zero); average must
  return a non-integer numeric result even for integer source fields; and an
  aggregation over a type-incompatible field (for example, summing a string)
  must fail with a structured type error (error codes per CONTRACT-002 /
  CONTRACT-007).

#### Surface Projections

- **AGG-04**: Aggregations must be exposed through the structured API and
  SDK along the shared handler path (FEAT-005); exact request/response
  payloads follow
  [CONTRACT-001](../../02-design/contracts/CONTRACT-001-http-api-surface.md)
  and the SDK aggregation helpers in
  [CONTRACT-009](../../02-design/contracts/CONTRACT-009-sdk-surface.md).
- **AGG-05**: A per-collection GraphQL aggregation field must be
  auto-generated for each registered collection, accepting filter, grouping,
  and aggregation-function arguments and returning total count plus grouped
  results; the field naming, argument shapes, and result types are normative
  in the Aggregations section of
  [CONTRACT-002](../../02-design/contracts/CONTRACT-002-graphql-surface.md).
- **AGG-06**: A per-collection MCP aggregation tool must be auto-generated
  for each registered collection, mirroring the GraphQL projection's
  parameters and results; the tool signature is normative in
  [CONTRACT-003](../../02-design/contracts/CONTRACT-003-mcp-surface.md).
- **AGG-07**: For the same subject, collection, filter, grouping, and
  functions, the structured API, GraphQL, and MCP projections must return
  identical aggregation results and identical structured errors.

#### Planner Integration

- **AGG-08**: Every aggregation must be expressible in the CONTRACT-007 V1
  Cypher subset and must compile through the unified planner's plan, limits,
  and policy enforcement. No separate aggregation engine, grammar, or
  execution path exists.
- **AGG-09**: Aggregations must be policy-safe: rows hidden from the caller
  by FEAT-029 policies are excluded before grouping and aggregation, so
  counts and totals never reveal hidden data (planner obligations per
  CONTRACT-007).
- **AGG-10**: When grouped or filtered fields carry secondary indexes
  (FEAT-013), the planner must use them to accelerate the aggregation;
  non-indexed aggregations fall back to scans within the planner's
  configured limits.
- **AGG-11**: Aggregations must see a consistent snapshot with the same read
  isolation as entity queries.

### Non-Functional Requirements

- **Latency**: Aggregation over 10K entities with an indexed grouping field
  completes in < 500ms p99.
- **Latency**: A total count without grouping completes in < 50ms.
- **Memory**: Aggregation working memory is bounded by the number of
  distinct group keys, not by entity count; a group-limit parameter caps the
  number of groups returned.

## Edge Cases and Error Handling

- **Grouping by a non-existent field**: All entities fall into a single
  null-labeled group.
- **Aggregation over an empty collection**: Returns a zero total count with
  empty groups.
- **Very high cardinality grouping**: Grouping by a unique field produces
  one group per entity; the group-limit parameter caps the number of groups
  returned.
- **Concurrent writes during aggregation**: The aggregation sees a
  consistent snapshot; writes that commit during execution may or may not be
  included depending on snapshot timing, never partially.
- **Policy-hidden rows**: Excluded from every group, count, and numeric
  aggregate — a caller cannot infer hidden rows from totals.

## Success Metrics

- 100% of supported aggregation shapes round-trip through the unified
  planner (no aggregation-only execution path) in conformance tests.
- 100% identical aggregation results and error shapes across structured
  API, GraphQL, and MCP on the shared parity fixture suite.
- Zero count/total leak findings for policy-hidden rows in the policy
  fixture suite.
- Indexed-grouping latency target met on the reference benchmark
  collection (10K entities).

## Constraints and Assumptions

- The read-side query language is read-only; aggregations are read
  projections and never mutate state.
- Aggregation capability is bounded by the CONTRACT-007 V1 Cypher subset;
  summaries that cannot be expressed there are out of scope until the
  language grows.
- Analytical workloads beyond operational summaries belong in downstream
  systems fed by change capture (PRD non-goal).

## Out of Scope

- **A standalone aggregation engine or grammar**: Aggregations exist only as
  projections of the unified Cypher planner (product-owner decision
  2026-06-10).
- **HAVING-style filters on aggregated results** (for example, "groups with
  count > 10"): deferred.
- **Window functions**: Running totals and rankings; deferred.
- **Cross-collection join aggregation**: Use change capture into a
  downstream analytical store.
- **Approximate aggregation**: Probabilistic distinct counts and similar;
  deferred.
